Two trends

So... There are two big changes on the web that are happening very suddenly.
The rise of mobile

The first is what Linus referred to: the rise of the multi-device web.

We now have more mobile devices than toothbrushes or pencils.

Commoditization of computing on a scale not seen since the PC boom of the 80s and 90s.

But this isn't just - same old thing, more computers.

With the change of devices comes a change in how we experience the web.

What we do on the web is changing fast.
The rise of media

The devices we're using to access the internet are naturally oriented to communication and consumption of media as well as the other kinds of computer interaction we've gotten used to.

So we have this second trend: the rise of audio, video and realtime communication.

And with the growth of mobile and media, the Web is evolving.

Offline media

53% of adults media multi-task while watching TV

Source: The reinvention of the 1950s living room, OFCOM August 2013

And very quickly, online media is transforming the way we consume media of all kinds.

For example, we still watch a lot of telly, but now most of us use a mobile device at the same time - and a lot of the time that means online video, audio and communication.

Online media

Video will be 80 to 90 percent of global internet consumer traffic by 2017

Source: Cisco Visual Networking Index: Forecast and Methodology, 2012–2017

In many countries, in fact, TV viewing is down and online viewing is up.

In China for example, in 2012 only 30% of households in Beijing watched TV, down from 70% in 2009.

Popular shows get audiences of 250 million or more.

Every second, nearly a million minutes of video content will cross the network in 2017.

The sum of all forms of video (TV, video on demand [VoD], Internet, and P2P) will be in the range of 80 to 90 percent of global consumer traffic by 2017.

It's not that we're all going to build the next YouTube, but we're coming to expect seamless audio, video and realtime communication in our web apps on all our devices.

A whole ecosystem of media APIs for the open Web.

Standardized, interoperable technologies that work across multiple platforms.

WebRTC

So - we have the ability to do great things with audio and video on the web.

For *streaming* audio and video in realtime, we have WebRTC...

...which is one the most ambitious and maybe disruptive web projects in many years.

WebRTC is a collaboration to build real time communication into web browsers.

It's really a response to the demand for realtime communication features in the open web - without plugins or proprietary codecs, free for users and developers to use.

This has been a big missing feature in the web platform...

...and is absolutely crucial for web apps on devices built for communication.

Mobile devices are the natural home for realtime communication.

Peer to peer

But perhaps the most radical thing about WebRTC is that its architecture is peer-to-peer - with a nice friendly fluffy cloud right in the middle.

Nothing is ever quite that simple, but keep this image in mind.

What do we need for RTC?

  • Acquire audio and video
  • Establish a connection between peers
  • Communicate audio and video
  • Communicate arbitrary data

So that's the vision for WebRTC. Now let's dig into how WebRTC works.

Three JavaScript APIs

  • MediaStreams (aka getUserMedia)
  • RTCPeerConnection
  • RTCDataChannel

Communicate Media Streams

→
getUserMedia
+
RTCPeerConnection

On the surface, the API is simple - get access to MediaStreams via getUserMedia, then plug them into a PeerConnection, and they will get sent to another WebRTC endpoint automatically. And when we receive media from the remote side, this goes into new MediaStreams that can be rendered in our web page.

RTCPeerConnection does a lot

  • Signal processing
  • Codec handling
  • Bandwidth management
  • Peer to peer communication
  • Security

...

Under the hood though, RTCPeerConnection is doing a lot - processing audio and video to remove noise, compressing the data using codecs, setting up the peer to peer pathway through NATs and firewalls, encrypting the data, ensuring we use the right amount of bandwidth...

WebRTC is built for security: encryption is mandatory for all components and, because it's built into the browser, WebRTC doesn't suffer the installation and update problems of plugins.

Sharefest

Sharefest allows us to share files using data channels.

It's straightforward to use: you just select a file and share a URL.

Sharefest then establishes a peer connection between each of the devices that open the URL.

Once that's done, the file can be shared directly, peer to peer, without going via an intermediary service.
Peer to peer file distribution with RTCDataChannel
peerCDN

peerCDN is even more radical.

This is a prototype peer-to-peer content distribution network.

To use peerCDN, all you have to do is to include a link to the peerCDN JavaScript file in your web page.

peerCDN will then attempt to get assets on the page from other connected peers. If that's not possible, assets will simply be downloaded from your web server as normal.

Web Audio

Now - enough about video. What about audio?

 
<audio src='chrome.mp3' />

This audio element is a thing of simple beauty.

Loads, decodes and plays audio - just like that!

Why Web Audio when we have <audio>?

  • Precise timing of multiple sounds
  • Audio pipeline for routing and processing
  • Analysis of audio data
  • <audio> can be used as input to Web Audio!

Where the audio element combines loading, decoding and playing into one black box, Web Audio gives you precise control over each of those steps, which means you can schedule sounds to sample accuracy. Timing is particularly important for games. In addition, Web Audio offers a powerful synthesis, routing and processing pipeline, and analysis tools.

demo of MIDI synth

and you can still use <audio> for streaming long files through web audio.

The Web Audio pipeline

  • Oscillators
  • 3D positioning of sounds
  • Fade-ins/fade-outs/sweeps
  • Time-based event scheduling
  • Doppler shift for moving sources
  • Frequency and waveform analysis
  • Acoustic environments: reverb, etc.
  • Waveshaping (non-linear distortion)
  • Dynamics processing (compression)
  • Filtering effects: radio, telephone, etc.
  • Low latency: input→speaker as low as 5ms
  • Integrated with WebRTC through MediaStream

We have lots of tools to easily do complex features, like fades, reverb, 3D positioning and more.

Web Audio is cross-browser!

  • Chrome desktop and Android — including gUM input
  • Firefox 25 desktop and Android!
  • Safari 6.0+ and iOS6+

DASH

Dynamic Adaptive Streaming over HTTP

  • Sample DASH player
  • DASH manifest file
  • dash.js
  • What is DASH?

EME demos use a JavaScript library to support MPEG DASH.

DASH allows adaptive streaming of media over HTTP, somewhat like Apple's HTTP Live Streaming.

You create multiple bitrate encodings of a video on the server then build a manifest file that describes these.

MSE implementations can then parse the manifest, download segments of video at an appropriate bitrate, and feed them to a video element when it gets hungry.

Adaptive delivery is crucial for dynamically coping with bandwidth constraints in the multi-device world.

Encrypted Media Extensions

  • Spec
  • YouTube demo
  • Netflix on Chrome OS
  • WebM encrypted + adaptive demo

EME is a JavaScript API, an extension of the HTMLMediaElement APIs, that enables web applications to interact with DRM systems, in order to allow playback of encrypted media.

EME can either use simple ClearKey encryption or an add-on DRM module, such as WideVine.

Browsers that support EME can detect that a video is encrypted when playback begins, then interact with Content Decryption Modules, CDMs, to obtain a key and play back decrypted content.

Captions and subtitles

simpl.info/track

simpl.info/track/audio

The track element allows you to display captions or subtitles from a VTT or SRT file. They're rendered over the video element.

You can also listen out for when cues change.

The track element is supported on Internet Explorer, and in Chrome for Android and desktop, and Safari on Mac OS.

Caption cues are displayed over the video, and we can also listen for cue events and get their contents.

In-band WebVTT: track data

To add subtitles or captions to a video, you simply add a track element.

The src points to a file in WebVTT format.

In-band WebVTT is also supported now by Chrome: WebVTT information packaged with a video file.

This makes it possible to store video with timed metadata of any kind.
